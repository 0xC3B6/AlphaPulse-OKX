# Legacy v0.1.3 Source Replay Design

Date: 2026-07-10

## Goal

Restore the last mature automatic `v0.1.3` paper strategy on the existing branch `codex/restore-v0.1.3`, whose current base is Git tag commit `0b65392`.

The restoration covers strategy source, new-run paper execution, and durable storage for all new strategy/account data. It does not recover or migrate old account balances, positions, trades, or PnL. The restored strategy starts from a fresh 10,000 USDT paper account.

## Confirmed Constraints

- Do not create another Git worktree.
- Do not modify `main` while restoring the strategy.
- Do not use the simplified Strategy Center runtime as the source of truth.
- Do not restore Strategy Center, v4 behavior, or old production account data.
- Port PostgreSQL and Redis infrastructure only as storage for the single restored v3 runtime.
- PostgreSQL is the sole authoritative state store; Redis is a rebuildable cache and is never used for recovery.
- Do not invent missing thresholds or signals from conversational summaries.
- Commit each recovered module immediately after its tests pass.
- Do not delete the current branches, worktrees, session log, or recovery evidence.

## Approaches Considered

### Selected: evidence replay and compatibility port

Recover complete new files from their creation patch plus all later successful patches. For existing files, reconstruct strategy-related changes against historical commit `907b9ec`, then port only those proven strategy changes onto the `0b65392` branch interfaces. Restore original tests before implementation and use later full source reads to verify replay output. Port the PostgreSQL/Redis infrastructure from commit `7be9e81` only after removing its Strategy Center and v4 coupling.

This preserves the requested frontend tag baseline while avoiding blind patch application across seven intervening commits.

### Rejected: treat tag v0.1.3 as the complete strategy

The tag contains only manual paper trading. It has no automatic strategy, time regime, market context, pattern execution, or automatic exits.

### Rejected: recreate behavior from remembered requirements

Natural-language requirements are useful for acceptance checks but are not sufficiently precise to recreate source thresholds and branch ordering. Exact patch and source evidence takes priority.

### Rejected: local JSON as the authoritative store

The original JSON file is simple but has weaker transaction, query, concurrency, and operational guarantees. A JSON export remains suitable for offline backup, but live restored-v3 state belongs in PostgreSQL.

## Recovery Classification

Every recovered component receives one classification in the committed recovery manifest:

- `exact_replay`: complete creation content and ordered successful patches are available;
- `verified_port`: historical source is available, but current tag interfaces require a mechanical adaptation verified by original tests;
- `behavior_only`: source evidence has a proven gap and only tests/runtime evidence survive.

No `behavior_only` implementation is written without first documenting the exact gap for user review.

## Components

### Pattern and feature model

Restore the recovered domain fields and pattern model used by v3, including W bottom, M top, sweep failure, pattern status, structure score, confirmation score, trade score, volume ratio, VWAP/ATR context, and multi-day movement tags.

Restore `backend/src/indicators/patterns.rs` from source evidence and its original tests before reconnecting it to runtime scoring.

### Time regime and market context

Restore `time_regime.rs` and `market_context.rs` from their creation patches, subsequent successful patches, and later full-file reads. Their recovered time windows, 0/4/8 penalties, labels, reversal windows, and market-context branches remain unchanged.

### Automatic v3 strategy

Restore `auto_strategy.rs` and `backend/tests/auto_strategy.rs` from source evidence. The runtime retains the recovered signal families:

- trend long/short;
- range long/short;
- pattern long/short;
- multiday extension reversal short;
- 24-hour mover long/short;
- time-regime penalties;
- market-context penalties;
- entry and exit tags/reasons.

Do not add a new global candidate-ranking algorithm unless it appears in the recovered source. Preserve recovered evaluation order and position-cap behavior.

### Paper execution and lifecycle

Restore the paper fields and execution behavior required by the recovered strategy:

| Setting | Value |
| --- | ---: |
| Initial equity | 10,000 USDT |
| Margin per automatic entry | 3% of current equity |
| Leverage | 20x |
| Maximum concurrent positions | 5 |
| Same-symbol positions | 1 |
| Default stop loss | -30% gross margin ROI |
| Default take profit | +40% gross margin ROI |
| Fee | 0.05% of notional per fill |
| Slippage | 0.02% adverse movement per fill |

Pattern structural stops may be tighter than the default stop and may not be wider. Valid W/M structural targets use the recovered 70% measured-move calculation and must satisfy the recovered reward-to-risk gate.

Manual closes remain manual. Automatic close reasons come from the actual trigger and are never inferred afterward from PnL sign.

### State integration

On a price update, the restored state performs:

1. mark the latest price;
2. evaluate every open position for stop, take profit, structural invalidation, or recovered expiry behavior;
3. execute and record triggered closes;
4. evaluate the recovered v3 entry logic;
5. execute eligible entries within the recovered account limits;
6. publish the updated paper snapshot.

The recovered gap fix executes a crossed stop or take profit at its stored trigger price before simulated slippage, not at an arbitrarily late mark price.

### Durable persistence architecture

PostgreSQL stores the single restored-v3 version, run, order intents, fills, open positions, closed positions, equity snapshots, event log, and a complete serialized account-state checkpoint. Redis stores only the latest API/dashboard snapshots with a finite TTL.

The restored runtime retains `version_code`, `run_id`, `strategy_build_id`, and `config_hash` fields even though Strategy Center is absent. These fields prevent a future implementation from silently reusing the same version number for different strategy code.

The existing schema from `7be9e81` is reused where it already represents generic trading data:

- `strategy_versions`;
- `strategy_runs`;
- `order_intents`;
- `fills`;
- `positions`;
- `closed_trades`;
- `equity_snapshots`;
- `event_log`;
- `app_state_snapshots`.

V4-only `risk_guard_events` is not part of the restored runtime. Existing deployments may retain the physical table until migration cleanup, but restored code neither reads nor writes it.

The `positions` schema is extended so open positions persist their protective and attribution state:

- `stop_loss`;
- `take_profit`;
- `expire_at_ms`;
- `primary_signal`;
- `strategy_reason`;
- `tags_json`;
- `config_hash`;
- `strategy_build_id`.

The complete account checkpoint contains balances, realized PnL, total fees, open positions, position metadata, trade/position history, next identifiers, run identity, strategy configuration, and the latest equity state.

### Persistence transaction boundaries

Every account-changing transition is committed to PostgreSQL in one transaction before it is published through the API or websocket:

- accepted automatic/manual entry: order intent, fill, open position, run totals, event log, and full checkpoint;
- rejected intent: order intent status and rejection reason;
- automatic/manual close: close intent, fill, closed trade, removal/closure of the open position, run totals, equity snapshot, event log, and full checkpoint;
- reset: old run closure and creation of the fresh run/checkpoint.

No accepted entry, close, fee, or realized-PnL change exists only in memory after the API/event reports success.

Mark prices and unrealized PnL are derived from live prices. Their latest checkpoint and equity curve are persisted once per completed market scan and immediately after every account-changing transition. A restart recalculates current unrealized PnL from restored positions and fresh prices.

### PostgreSQL failure policy

Production and restored local runs require PostgreSQL. If connection or schema initialization fails, automatic strategy processing does not start.

If a PostgreSQL transaction fails while running, the runner enters `persistence_paused`, publishes a persistence-health error, stops consuming new strategy decisions, and retains only the last committed account state. It retries the database connection and resumes from the committed checkpoint. It never reports an uncommitted trade as successful.

### Redis cache policy

Redis keys contain only the latest dashboard, paper-account, positions, history, and strategy-health snapshots. Cache writes happen after the PostgreSQL transaction commits. Redis failure does not roll back a committed PostgreSQL trade, does not stop protective strategy processing, and does not affect restart recovery. Missing or stale cache entries are rebuilt from PostgreSQL and current market data.

### Startup and restart recovery

Startup performs this sequence before the scanner starts:

1. require and connect to PostgreSQL;
2. apply idempotent schema migrations;
3. load the latest restored-v3 checkpoint;
4. validate `version_code`, `strategy_build_id`, and `config_hash`;
5. restore identifiers, balances, fees, positions, history, and protective levels;
6. refresh marks from current market prices;
7. rebuild Redis cache;
8. start automatic strategy processing.

If no restored-v3 checkpoint exists, the runtime creates and commits a new 10,000-USDT run before scanning. A checkpoint with a different build identifier is rejected instead of being silently loaded.

### Existing-data cleanup

Old data is not imported into the restored account. Before the first restored run, an administration operation exports existing simplified-v3/v4 strategy rows to offline JSON, verifies file hashes, removes those strategy rows and stale Redis keys, and commits the new empty restored-v3 run. Unrelated Radar and Macro data is left intact.

### Frontend

Keep the `0b65392` frontend baseline. Restore only the types and presentation necessary to show automatic v3 positions, entry reason, signal tags, stop/take-profit metadata, close reason, fees, and strategy identity. Do not add Strategy Center or v4 UI.

## Replay Procedure

1. Verify the session-log hash before extraction.
2. Record the historical base commit and all relevant successful patch event IDs in a committed manifest.
3. Restore original tests first; confirm they fail for missing legacy modules.
4. Reconstruct new source files from their creation content and ordered successful patches.
5. Port proven strategy hunks in existing files to the tag interfaces.
6. Port single-v3 PostgreSQL/Redis persistence and write persistence tests.
7. Re-run recorded formatting commands.
8. Compare reconstructed files with later overlapping full source reads.
9. Run the narrow original tests after each module.
10. Commit each passing module separately.
11. Run all backend and frontend tests and production builds.
12. Run a PostgreSQL restart-recovery integration test with Redis emptied.
13. Export and clean old strategy data, then commit a fresh 10,000-USDT restored run.
14. Restart the existing local backend/frontend services from this branch.

## Error Handling

- Session hash mismatch: stop without changing strategy files.
- Patch marked failed in the log: do not replay it.
- Patch context mismatch: compare historical base and tag versions; port the proven hunk manually and add a characterization test.
- Missing final source coverage: classify the component `behavior_only` and stop for user review.
- Recovered original test failure: do not continue to the next component.
- PostgreSQL unavailable at startup: do not start automatic strategy processing.
- PostgreSQL transaction failure while running: enter `persistence_paused` and expose a health error without publishing an uncommitted transition.
- Redis unavailable: continue from PostgreSQL and rebuild cache after Redis recovers.
- Live market produces no immediate signal: do not create fake positions; demonstrate automatic behavior with deterministic test fixtures and leave the live scanner running.

## Verification

### Source verification

- Recovery manifest records every replayed source event and content hash.
- Replaying the manifest produces stable hashes for exact-replay files.
- All adaptations are limited to interface compatibility and have tests.

### Strategy verification

- Original auto-strategy tests compile and pass.
- All recovered signal families retain their primary signal and reason text.
- A 10,000-USDT fixture opens a 300-USDT-margin, 20x position.
- A sixth concurrent position is rejected.
- A duplicate-symbol entry is rejected.
- Every automatic position has executable stop and take-profit levels.
- Long and short stop/take-profit boundaries close automatically.
- Trigger-price gap behavior is covered by a regression test.
- Fees and slippage are applied on entry and exit.

### Full verification

- `cargo fmt --check` passes.
- all backend tests pass;
- backend production build passes;
- all frontend tests pass;
- frontend production build passes;
- Git diff contains no v4 or Strategy Center implementation;
- local frontend and backend start from this branch;
- `/api/health` succeeds and live market data connects;
- the fresh paper snapshot identifies restored automatic `v0.1.3`.

### Persistence verification

- PostgreSQL contains the restored version/build/config identity and one fresh run.
- Accepted/rejected intents, fills, open positions, protective levels, closes, fees, PnL, history, and equity snapshots are queryable from PostgreSQL.
- A forced failure in the middle of a transition rolls back the whole transition.
- Restarting the backend restores the same balance, identifiers, fees, positions, stops, take profits, and history.
- Clearing Redis before restart does not lose data; the cache is rebuilt from PostgreSQL.
- Starting with PostgreSQL unavailable does not run the automatic strategy.
- Redis failure does not prevent a PostgreSQL-committed transition.

## Completion Criteria

The strategy is restored only when:

1. source evidence and recovery classification are committed;
2. recovered original tests pass;
3. account sizing, leverage, signals, exits, fees, and slippage match the recovered v3 evidence;
4. every accepted or rejected strategy decision and every account transition is durably represented in PostgreSQL;
5. backend restart and Redis-loss tests restore the same committed paper state;
6. the local app runs the automatic strategy from a fresh persisted account;
7. every recovered module is committed and no strategy source remains only in an uncommitted worktree;
8. a recovery candidate Git tag is ready to create after user inspection.
