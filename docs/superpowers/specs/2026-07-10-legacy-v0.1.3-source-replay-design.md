# Legacy v0.1.3 Source Replay Design

Date: 2026-07-10

## Goal

Restore the last mature automatic `v0.1.3` paper strategy on the existing branch `codex/restore-v0.1.3`, whose current base is Git tag commit `0b65392`.

The restoration covers strategy source and new-run paper execution only. It does not recover or migrate old account balances, positions, trades, or PnL. The restored strategy starts from a fresh 10,000 USDT paper account.

## Confirmed Constraints

- Do not create another Git worktree.
- Do not modify `main` while restoring the strategy.
- Do not use the simplified Strategy Center runtime as the source of truth.
- Do not restore v4, PostgreSQL, Redis, or old production data in this phase.
- Do not invent missing thresholds or signals from conversational summaries.
- Commit each recovered module immediately after its tests pass.
- Do not delete the current branches, worktrees, session log, or recovery evidence.

## Approaches Considered

### Selected: evidence replay and compatibility port

Recover complete new files from their creation patch plus all later successful patches. For existing files, reconstruct strategy-related changes against historical commit `907b9ec`, then port only those proven strategy changes onto the `0b65392` branch interfaces. Restore original tests before implementation and use later full source reads to verify replay output.

This preserves the requested frontend tag baseline while avoiding blind patch application across seven intervening commits.

### Rejected: treat tag v0.1.3 as the complete strategy

The tag contains only manual paper trading. It has no automatic strategy, time regime, market context, pattern execution, or automatic exits.

### Rejected: recreate behavior from remembered requirements

Natural-language requirements are useful for acceptance checks but are not sufficiently precise to recreate source thresholds and branch ordering. Exact patch and source evidence takes priority.

## Recovery Classification

Every recovered component receives one classification in the committed recovery manifest:

- `exact_replay`: complete creation content and ordered successful patches are available;
- `verified_port`: historical source is available, but current tag interfaces require a mechanical adaptation verified by original tests;
- `behavior_only`: source evidence has a proven gap and only tests/runtime evidence survive.

No `behavior_only` implementation is written without first documenting the exact gap for user review.

## Components

### Pattern and feature model

Restore the recovered domain fields and pattern model used by v3, including W bottom, M top, sweep failure, pattern status, structure score, confirmation score, trade score, volume ratio, VWAP/ATR context, and multi-day movement tags.

Restore `backend/src/indicators/patterns.rs` from source evidence and its original tests before reconnecting it to runtime scoring.

### Time regime and market context

Restore `time_regime.rs` and `market_context.rs` from their creation patches, subsequent successful patches, and later full-file reads. Their recovered time windows, 0/4/8 penalties, labels, reversal windows, and market-context branches remain unchanged.

### Automatic v3 strategy

Restore `auto_strategy.rs` and `backend/tests/auto_strategy.rs` from source evidence. The runtime retains the recovered signal families:

- trend long/short;
- range long/short;
- pattern long/short;
- multiday extension reversal short;
- 24-hour mover long/short;
- time-regime penalties;
- market-context penalties;
- entry and exit tags/reasons.

Do not add a new global candidate-ranking algorithm unless it appears in the recovered source. Preserve recovered evaluation order and position-cap behavior.

### Paper execution and lifecycle

Restore the paper fields and execution behavior required by the recovered strategy:

| Setting | Value |
| --- | ---: |
| Initial equity | 10,000 USDT |
| Margin per automatic entry | 3% of current equity |
| Leverage | 20x |
| Maximum concurrent positions | 5 |
| Same-symbol positions | 1 |
| Default stop loss | -30% gross margin ROI |
| Default take profit | +40% gross margin ROI |
| Fee | 0.05% of notional per fill |
| Slippage | 0.02% adverse movement per fill |

Pattern structural stops may be tighter than the default stop and may not be wider. Valid W/M structural targets use the recovered 70% measured-move calculation and must satisfy the recovered reward-to-risk gate.

Manual closes remain manual. Automatic close reasons come from the actual trigger and are never inferred afterward from PnL sign.

### State integration

On a price update, the restored state performs:

1. mark the latest price;
2. evaluate every open position for stop, take profit, structural invalidation, or recovered expiry behavior;
3. execute and record triggered closes;
4. evaluate the recovered v3 entry logic;
5. execute eligible entries within the recovered account limits;
6. publish the updated paper snapshot.

The recovered gap fix executes a crossed stop or take profit at its stored trigger price before simulated slippage, not at an arbitrarily late mark price.

### Fresh local persistence

Old `.data/paper_state.json` is not recovered. If the recovered runtime expects local JSON persistence, it creates a new ignored state file for the fresh account. Historical rows or balances are never synthesized.

### Frontend

Keep the `0b65392` frontend baseline. Restore only the types and presentation necessary to show automatic v3 positions, entry reason, signal tags, stop/take-profit metadata, close reason, fees, and strategy identity. Do not add Strategy Center or v4 UI.

## Replay Procedure

1. Verify the session-log hash before extraction.
2. Record the historical base commit and all relevant successful patch event IDs in a committed manifest.
3. Restore original tests first; confirm they fail for missing legacy modules.
4. Reconstruct new source files from their creation content and ordered successful patches.
5. Port proven strategy hunks in existing files to the tag interfaces.
6. Re-run recorded formatting commands.
7. Compare reconstructed files with later overlapping full source reads.
8. Run the narrow original tests after each module.
9. Commit each passing module separately.
10. Run all backend and frontend tests and production builds.
11. Reset to a fresh 10,000 USDT local paper account.
12. Restart the existing local backend/frontend services from this branch.

## Error Handling

- Session hash mismatch: stop without changing strategy files.
- Patch marked failed in the log: do not replay it.
- Patch context mismatch: compare historical base and tag versions; port the proven hunk manually and add a characterization test.
- Missing final source coverage: classify the component `behavior_only` and stop for user review.
- Recovered original test failure: do not continue to the next component.
- Live market produces no immediate signal: do not create fake positions; demonstrate automatic behavior with deterministic test fixtures and leave the live scanner running.

- all backend tests pass;
- backend production build passes;
- all frontend tests pass;

- Git diff contains no v4 or Strategy Center implementation;

- `/api/health` succeeds and live market data connects;
- the fresh paper snapshot identifies restored automatic `v0.1.3`.

## Completion Criteria

The strategy is restored only when:

1. source evidence and recovery classification are committed;
2. recovered original tests pass;
3. account sizing, leverage, signals, exits, fees, and slippage match the recovered v3 evidence;
4. the local app runs the automatic strategy from a fresh account;
5. every recovered module is committed and no strategy source remains only in an uncommitted worktree;
6. a recovery candidate Git tag is ready to create after user inspection.
